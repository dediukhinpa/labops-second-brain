# AGENT.md — deploy instructions for Claude Code

You are a Claude Code agent. Your job is to deploy a Second Brain system end-to-end on the user's infrastructure. Depending on which **path** the user picks, you will deploy either:

- **Path A (Minimal):** the shared brain on a fresh Ubuntu 22.04 LTS VPS plus a single local Telegram inbox-agent. ~30 minute install.
- **Path B (Full stack):** everything in Path A, PLUS one or more personal Claude Code agent workspaces under `~/.claude-lab/<agent-id>/.claude/` generated by `agent-template/install.sh`. Each workspace is a complete Claude Code agent with layered memory (hot/warm/cold), MCP recall into the shared brain, hooks, and a skills bundle. ~60–90 minute install.

Read this entire file before doing anything. The user has cloned this repo and is now handing it to you. They expect a working system at the end, not a half-deployed mess.

---

## 1. Role and scope

You are an autonomous deployer. You will:

1. Confirm with the user whether they want **Path A** or **Path B** (use the decision table in section 2).
2. Verify prerequisites and ask the user for every input you cannot guess.
3. Push the repo to the VPS, run the install script, verify health.
4. Issue agent tokens, configure the local inbox-agent, set up cron jobs.
5. **(Path B only)** For each personal agent the user wants, run `agent-template/install.sh`, issue a per-agent token, wire the workspace's `.mcp.json`, set up per-workspace cron jobs.
6. Run an end-to-end smoke test that proves a Telegram message lands in the shared brain and is retrievable through recall — and (Path B) that each personal agent workspace opens cleanly and can recall from the brain.
7. Report a verifiable done-state to the user.

You will NOT:

- Make architectural changes. The architecture is fixed; do not "improve" the recall ranking, the dual-write pattern, the auth model, or the vault layout.
- Touch `agent-template/` source files. The template directory is generated by a separate process; you only **use** its `install.sh` and **document** the workspaces it produces.
- Run `sudo` outside `scripts/install.sh`.
- Push anything to any public git remote.
- Skip checks. If a step fails, you stop and tell the user.

The system you deploy has up to four layers (the last is Path B only):

1. **Shared brain (VPS):** 3 MCP services (memory write / memory_router read / agent_router event bus) backed by Postgres + pgvector, plus an ingest worker that embeds new vault files. The vault is 12 folders of markdown.
2. **Inbox-agent (local):** a Telegram bot the user forwards content to (links, voice notes, screenshots). It dual-writes — once to a local `raw/` folder for resilience, once to the shared brain via the memory MCP. Cron jobs compile raw into structured notes daily and send a digest.
3. **Skills bundle (local):** optional ingestion skills the inbox-agent (and any personal agent) can invoke per content type (YouTube transcripts, Instagram captions, X threads, voice-to-text, generic markdown).
4. **Personal agent workspaces (local, Path B):** one or more `~/.claude-lab/<agent-id>/.claude/` workspaces produced by `agent-template/install.sh`. Each has its own SOUL (CLAUDE.md), rules, layered memory (handoff.md → decisions.md → MEMORY.md/LEARNINGS.md/TOOLS.md), Stop/SessionStart/PreCompact hooks, and an `.mcp.json` wired to the shared brain. Multiple workspaces share **one** brain — that is the point.

## 3. What you are building (combined architecture)

```
                    +----------------------------------------------+
                    |              User's workstation              |
                    |                                              |
                    |  +--------------------+                      |
                    |  |  inbox-agent       |                      |
                    |  |  (Telegram bot)    |                      |
                    |  +---------+----------+                      |
                    |            |                                 |
                    |            | dual-write                      |
                    |            |  /        \                     |
                    |            | local raw/  Bearer              |
                    |            |              |                  |
                    |  +---------v----------+   |                  |
                    |  | Personal agents    |   |  (Path B only)   |
                    |  | (~/.claude-lab/    |   |                  |
                    |  |   <agent-id>/      |   |                  |
                    |  |   .claude/)        |   |                  |
                    |  | + hot/warm/cold    |   |                  |
                    |  |   memory layers    |   |                  |
                    |  | + Stop hook        |   |                  |
                    |  | + SessionStart     |   |                  |
                    |  | + PreCompact       |   |                  |
                    |  | + .mcp.json        |---+                  |
                    |  +--------------------+   |                  |
                    |                           |                  |
                    +---------------------------|------------------+
                                                |
                                            HTTPS or
                                            Tailscale
                                                |
              +---------------------------------v---------------+
              |              VPS (Ubuntu 22.04)                 |
              |                                                 |
              |   memory_mcp         :5001  (127.0.0.1 only)   |
              |   memory_router_mcp  :5002  (127.0.0.1 only)   |
              |   agent_router_mcp   :5000  (127.0.0.1 only)   |
              |                                                 |
              |   ingest-worker (systemd)                       |
              |     watches embedding_jobs --> embeds chunks    |
              |                                                 |
              |   Postgres 16 + pgvector                        |
              |     agent_tokens (one row per agent),           |
              |     audit_log, delivery_outbox,                 |
              |     documents (body + body_tsv) +               |
              |     chunks (content + embedding vector(1024))   |
              |                                                 |
              |   vault/  (12 markdown folders)                 |
              +-------------------------------------------------+
```

**The 3 MCP services:**

| Service | Port | Purpose | Scopes it writes |
|---|---|---|---|
| `memory_mcp` | 5001 | Write tools: create decision, error-pattern, personal note, project note, external note, daily log entry | daily, decisions, external, knowledge, error-patterns, personal, projects, inbox |
| `memory_router_mcp` | 5002 | Read tools: recall (hybrid search), recent, related, get-by-id, stats | none (read-only) |
| `agent_router_mcp` | 5000 | Inter-agent event bus: notify, ack, list-pending, broadcast | none (writes to `outbox` table only) |

**Auth:** each agent gets a Bearer token. Token sha256 is stored in `agent_tokens.token_sha256`. Each request's `Authorization: Bearer <token>` header is captured by `AuthCaptureMiddleware` and the token resolves to an agent identity with scoped write permissions. No header → 401. No silent fallback. Ever. **In Path B, each personal agent workspace gets its own distinct Bearer** — never share tokens between agents.

**Recall:** hybrid search combines (a) pgvector cosine over FastEmbed `multilingual-e5-large` (1024-dim) embeddings stored per chunk in `chunks.embedding` and (b) Postgres full-text search over `chunks.content_tsv` / `documents.body_tsv`. Results are fused with Reciprocal Rank Fusion (RRF), then re-ranked by temporal decay and a source_weights map (e.g. `error-patterns` × 3.0, `inbox` × 0.5).

## 6. Deploy sequence — Path A (10 steps)

These 10 steps bring up the shared brain and the inbox-agent. Path B continues from step 11 in section 7.

Execute these in order. After each step, verify the success condition before moving on. Do not skip a step. Do not run them in parallel.

### Step 1: Verify VPS is Ubuntu 22.04 LTS

```bash
ssh <USER>@<VPS_IP> "lsb_release -a"
```

Expect `Distributor ID: Ubuntu` and `Release: 22.04`. If the user's VPS is on a different release (20.04, 24.04, Debian, etc.) — STOP. Tell the user: "This installer is tested only on Ubuntu 22.04 LTS. Please provision a 22.04 VPS and re-run."

### Step 2: Push the repo to the VPS

Use `rsync` over SSH. Target directory `/opt/second_brain/` (the default `INSTALL_DIR`). If the user picked a different `INSTALL_DIR`, substitute.

```bash
rsync -avz --delete \
  --exclude '.git' \
  --exclude '.venv' \
  --exclude '__pycache__' \
  --exclude 'vault-template/**/.gitkeep' \
  --exclude 'agent-template' \
  ./ <USER>@<VPS_IP>:/opt/second_brain/
```

Note: `agent-template/` is excluded from the VPS push — it is a **local-only** generator. The VPS does not need it.

Verify with `ssh <USER>@<VPS_IP> "ls /opt/second_brain/"` — you should see `services/`, `migrations/`, `scripts/`, etc.

### Step 3: Run the install script on the VPS

```bash
ssh <USER>@<VPS_IP> "cd /opt/second_brain && sudo bash scripts/install.sh"
```

This is idempotent and takes 10–15 minutes. It will:

- `apt update` and install Python 3.11, Postgres 16, pgvector
- create a system user `second_brain`
- create the Postgres database `second_brain` and user `second_brain` with a generated password
- run migrations from `migrations/`
- set up a Python venv and `pip install` requirements
- install systemd unit files from templates in `systemd/`
- generate the first admin agent token and print it ONCE
- start all 4 services (`second_brain-memory-mcp`, `second_brain-memory_router-mcp`, `second_brain-agent_router-mcp`, `second_brain-ingest-worker`)

Read the script's output carefully. The admin token is printed only once — capture it. If the script exits non-zero, do not proceed. Look at the last 50 lines for the actual error, then check `docs/troubleshooting.md`.

### Step 4: Verify all 4 services are healthy

```bash
ssh <USER>@<VPS_IP> "systemctl status second_brain-memory-mcp second_brain-memory_router-mcp second_brain-agent_router-mcp second_brain-ingest-worker --no-pager"
```

All four must be `active (running)`. If any is `failed` or `activating (auto-restart)`, pull logs:

```bash
ssh <USER>@<VPS_IP> "journalctl -u <service-name> -n 100 --no-pager"
```

Common failures and fixes are in `docs/troubleshooting.md`. Do not guess — read logs first.

### Step 5: Run the smoke test from the VPS itself

```bash
ssh <USER>@<VPS_IP> "cd /opt/second_brain && bash scripts/smoke-test.sh"
```

This script:

- queries each MCP service's HTTP root, expects 200 with service banner
- sends a `tools/list` JSON-RPC call to each `/mcp` endpoint with the admin Bearer token, expects a tool list
- inserts a test row into `agent_tokens` for `inbox-agent`, queries it back, deletes it (auth round-trip)
- exits 0 on full pass

If smoke-test fails, do not move on. The most common cause is `AuthCaptureMiddleware` not loaded — verify `services/memory_mcp/server.py` and `services/memory_router_mcp/server.py` both have the middleware (they should, in this distro; if not, the repo is broken).

### Step 6: Issue agent tokens

You need at minimum two tokens for Path A:

- **coordinator token** — for the user's main Claude Code agent that recalls and writes decisions. Full scopes: `daily, decisions, external, knowledge, error-patterns, inbox`.
- **inbox-agent token** — restricted to the scopes inbox-agent writes: `decisions, external, knowledge, inbox`.

Issue them with:

```bash
ssh <USER>@<VPS_IP> "cd /opt/second_brain && python scripts/issue-agent-token.py --agent coordinator-agent --scopes 'daily,decisions,external,knowledge,error-patterns,inbox'"
ssh <USER>@<VPS_IP> "cd /opt/second_brain && python scripts/issue-agent-token.py --agent inbox-agent --scopes 'decisions,external,knowledge,inbox'"
```

Each command prints the token ONCE to stdout. Capture both. Hand them to the user with the explicit instruction to save them in a password manager. Never write them into the repo, the `.env.example`, or any committed file.

If the user loses a token, the recovery path is `python scripts/issue-agent-token.py --agent <name> --revoke` (marks all live rows as revoked) followed by a fresh `--scopes '...'` issue.

For **Path B**, you will issue one **additional** token per personal agent in step 14 (section 7). Each personal agent gets its own — never reuse a token across agents.

### Step 7: Local setup — install inbox-agent on the user's workstation

On the user's local machine (not the VPS):

```bash
bash scripts/install-local.sh
```

This script:

- creates `${INBOX_AGENT_HOME}` (default `~/.claude-lab/inbox-agent`)
- copies `inbox-agent/` skeleton into it
- writes `${INBOX_AGENT_HOME}/.claude/.mcp.json` from `inbox-agent/config/.mcp.json.template`, filling in the VPS URL (`MCP_HOST`) and the inbox-agent Bearer token (`AGENT_TOKEN`)
- writes a `.env` with the optional API keys the user provided
- `chmod 600` on `.env` and `.mcp.json`
- verifies the inbox-agent can reach the VPS's `/memory_router/mcp` endpoint with its token

If the local reachability check fails, the most likely cause is firewall, missing DNS, or the user's machine not having Tailscale up (if Tailscale-only). Fix that before proceeding.

### Step 8: BotFather flow + run the bot daemon

If the user did not provide a bot token in section 5.1 question 4, walk them through this now:

1. Open `@BotFather` in Telegram.
2. Send `/newbot`.
3. Choose a name (display, can be anything).
4. Choose a username (must end in `bot`, e.g. `my_inbox_bot`).
5. Copy the token BotFather returns (format `123456:ABC-DEF...`).
6. Send `/setprivacy` → choose the bot → `Enable` (recommended — bot only sees DMs and replies).

Write the token into `${INBOX_AGENT_HOME}/.env` as `TELEGRAM_BOT_TOKEN=` and the digest token file at `${INBOX_AGENT_HOME}/secrets/telegram-bot-token` (mode 600).

Set the bot's allowlist via `BOSS_CHAT_ID` (the user's own Telegram user_id) in `${INBOX_AGENT_HOME}/.env` — the bot ignores everyone else.

Now run the bot daemon. The repo ships `inbox-agent/bot.py` (a thin python-telegram-bot polling daemon that calls `hooks/save-to-raw.sh` on every inbound message). Install its deps and start it:

```bash
"${INBOX_AGENT_HOME}/.venv/bin/pip" install -r "${INBOX_AGENT_HOME}/requirements.txt"
nohup "${INBOX_AGENT_HOME}/.venv/bin/python" "${INBOX_AGENT_HOME}/bot.py" \
  >> "${INBOX_AGENT_HOME}/logs/bot.log" 2>&1 &
```

For a long-running setup, wire `bot.py` into systemd (Linux) or launchd (macOS) instead of `nohup`. A reference unit at `${INBOX_AGENT_HOME}/inbox-agent.service` ships with the install — adapt and place it under `/etc/systemd/system/` or `~/Library/LaunchAgents/`.

Verify the bot is up:

```bash
curl -sS "https://api.telegram.org/bot${TELEGRAM_BOT_TOKEN}/getMe" | jq .
tail -20 "${INBOX_AGENT_HOME}/logs/bot.log"
```

`getMe` returns `ok: true` with the bot's username. `bot.log` should show `Application started`.

### Step 9: Install crontab

```bash
crontab inbox-agent/scripts/crontab.example
crontab -l
```

The crontab installs:

- `*/15 * * * *` — `compile.sh`: process new entries from local `raw/` into structured notes, dual-write to memory MCP
- `0 9 * * *` — `daily-digest.sh`: build yesterday's digest, send to the user's Telegram (09:00 UTC)

If the user already has a crontab, do not blow it away — append. Use `(crontab -l 2>/dev/null; cat inbox-agent/scripts/crontab.example) | crontab -`.

### Step 10: End-to-end smoke test

This is the only step that proves the Path A system actually works.

1. From the user's Telegram, send a YouTube URL (or any URL) directly to the inbox-agent bot.
2. Within ~2 seconds the bot should reply with a short ack ("Got it" — see `inbox-agent/bot.py` for the exact text).
3. The hook (`save-to-raw.sh`) runs synchronously inside the bot: it writes the raw markdown into `${INBOX_AGENT_HOME}/raw/...` AND posts to `memory_mcp.create_external_note` over the bearer in `.claude/.mcp.json`.
4. From a fresh Claude Code agent context configured with the coordinator-agent token (use `${INBOX_AGENT_HOME}/.claude/.mcp.json` as reference), call:

   ```
   recall.recent(scope="external", limit=5)
   ```

5. The just-sent URL must appear in the results, with `agent: inbox-agent` and a recent `created_at` (within the last 60 seconds).

If the URL appears → Path A is DONE. If the user picked Path A only, go straight to section 10 (Done condition). If the user picked Path B, continue with section 7.

If the URL does NOT appear, debug in this order:

1. Check the bot got the message and replied: `tail -f ${INBOX_AGENT_HOME}/logs/bot.log` and `tail -f ${INBOX_AGENT_HOME}/logs/save-to-raw.log`.
2. Check `raw/` has the new file: `ls -la ${INBOX_AGENT_HOME}/raw/ | head`.
3. Check the dual-write attempted: `grep memory ${INBOX_AGENT_HOME}/logs/save-to-raw.log | tail`. A successful write logs `memory-mcp create_external_note 200`. A failure logs the curl HTTP code and body.
4. On the VPS: `journalctl -u second_brain-memory-mcp -n 200 --no-pager` and `journalctl -u second_brain-ingest-worker -n 200 --no-pager`.
5. Verify in Postgres: `psql -U second_brain -d second_brain -c "SELECT path, agent, created_at FROM documents WHERE scope='external' ORDER BY created_at DESC LIMIT 5;"`.

The fault is almost always (a) wrong bearer token in `${INBOX_AGENT_HOME}/.claude/.mcp.json`, (b) wrong VPS URL there, (c) firewall blocking outbound 443, or (d) `bot.py` not running. Address those before digging deeper.

### Step 16: Set up per-workspace crontab entries

Each workspace ships its own memory-rotation scripts. They need cron entries appended (not replaced) to the user's crontab. For each workspace:

```bash

# memory rotation for <agent-id>

30 3 * * * /bin/bash -lc "$WORKSPACE/scripts/rotate-warm.sh      >> $WORKSPACE/logs/rotate-warm.log 2>&1"
```

## 8. RED operations — STOP and ask the user explicitly

These operations require an explicit "yes, proceed" from the user before you run them. Show the exact command you are about to run, explain the impact, wait for confirmation.

- Any `sudo` outside `scripts/install.sh`, `scripts/install-vps.sh`, and `agent-template/install.sh`. The install scripts contain all expected privilege escalation. Anything else is a surprise.
- `rm -rf` anywhere. If you need to clean up a partial install (including a half-created workspace), ask first.
- Schema changes to the second_brain database beyond what `migrations/` applies. No ad-hoc `ALTER TABLE`. If a migration is needed, write a new file `migrations/00X_<name>.sql` and propose it to the user first.
- Exposing services on a public IP without TLS. By default, all MCP services bind `127.0.0.1` only — nothing is public-facing. If the operator needs remote access, the supported options are Tailscale (private network, no public TLS required) or a self-managed reverse proxy with TLS (operator's responsibility — see docs/architecture.md). Plain HTTP on `0.0.0.0` is forbidden regardless of approach.
- Pushing anything to any public GitHub repository. The user controls publication. Even `git remote add` should be confirmed.
- Modifying anything in `/etc/` outside `install.sh`'s documented changes (systemd units in `/etc/systemd/system/second_brain-*.service`). No editing `/etc/postgresql/`, `/etc/ssh/`, `/etc/sudoers`, etc.
- Restarting `second_brain-*` services in production-like conditions (i.e. when the user has been using the system for more than a day). The user may be relying on them.
- Deleting or rotating an agent token without first checking the user is not actively using it.
- **Deleting an existing personal agent workspace.** `rm -rf ~/.claude-lab/<agent-id>` removes the agent's local memory (HOT, WARM, COLD layers, learnings). Confirm first that the user wants this and that they have a backup if the workspace had useful history.
- **Sharing a Bearer token between two agents.** Never. Even if "they do similar things" — they are separate identities for audit.

When in doubt, ask. Asking is cheap. Re-installing because you broke something is expensive.

## 9. Guardrails — never

These are non-negotiable. There is no scenario where bypassing them is correct.

- Never skip `scripts/sanitize-check.sh` before any commit the user makes. If the user asks you to commit changes back to a fork, run sanitize-check first.
- Never use API keys you found in the repo. `.env.example` is a template — every value is a placeholder. Real keys come from the user, are entered into `${INBOX_AGENT_HOME}/.env` (chmod 600) or `~/.claude-lab/<agent-id>/.claude/.mcp.json` (chmod 600), and are never echoed back to the user in a message.
- Never push secrets to git. Not in `.env.example`, not in commit messages, not in `docs/`. If the user accidentally commits a secret, walk them through revocation immediately (rotate the underlying credential first, then `git rm` and force-push, never the reverse).
- **Never share Bearer tokens between agents.** Each personal agent in Path B has its own token row in `agent_tokens`. Reusing tokens breaks audit attribution and makes revocation a lockout instead of a scalpel.
- Never restart a service without checking its current logs first. `journalctl -u <service> -n 50` takes 1 second and saves you from restarting a service that was about to recover.
- Never silently work around a failed check. If `smoke-test.sh` fails, you do not patch around it. You diagnose, fix, and re-run.
- Never assume retry will fix it. If a `pip install` fails the first time, the second time will fail too unless you fix the underlying issue. Read the error.
- Never modify files inside `agent-template/` itself. The template is the source of truth for new workspaces; if you "fix" it inline, the next agent install gets a divergent skeleton.

## 10. When you encounter errors

There is a strict order to error handling. Do not skip steps.

1. **First — read `docs/troubleshooting.md`.** It catalogs the most common failures (401s, embedding pipeline stalls, bot not responding, agent-template install quirks) with the exact fix for each.

2. **Second — pull service logs.** For systemd services on the VPS:

   ```bash
   ssh <USER>@<VPS_IP> "journalctl -u second_brain-memory-mcp -n 200 --no-pager"
   ssh <USER>@<VPS_IP> "journalctl -u second_brain-memory_router-mcp -n 200 --no-pager"
   ssh <USER>@<VPS_IP> "journalctl -u second_brain-agent_router-mcp -n 200 --no-pager"
   ```

   For the local inbox-agent: `tail -100 ${INBOX_AGENT_HOME}/logs/*.log`.

   For a Path B personal agent workspace: `tail -100 ~/.claude-lab/<agent-id>/.claude/logs/*.log`.

3. **Third — report to the user.** When you cannot resolve the error from the docs + logs, write a clear report with:

   - the exact command you ran
   - the exact error output (last 30 lines, fenced)
   - your hypothesis about the cause
   - the specific next step you propose (and why)

   Never silently work around a failed check or pretend a step succeeded. Never "try again" without changing anything.
